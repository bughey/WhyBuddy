# 设计文档：任务自动驾驶驾驶舱信息架构

## 设计概述

本 spec 定义任务自动驾驶主界面的信息架构，不直接定义最终视觉风格、具体组件代码或 3D 场景表现，而是确定用户在执行复杂任务时应该首先看到什么、在哪里理解路线、在哪里接管、在哪里查看证据。

该设计基于现有 `mission-first` 与 `Mission Runtime` 底座，目标是把已有的任务、执行、回放、审计、人机协同能力收敛为一个稳定的“驾驶舱”主视图，而不是继续分散在聊天页、任务详情页、回放页和审计页之间。

## 核心设计

### 1. 三栏结构

#### 1.1 左侧：目标与路线栏

左侧用于回答三个问题：

1. 我要去哪里
2. 系统打算怎么去
3. 现在已经走到哪里

建议承载以下信息块：

- 目的地卡片
  - 任务一句话目标
  - 目标摘要
  - 成功标准
  - 约束条件
- 路线卡片
  - 当前推荐路线
  - 候选路线
  - 当前阶段
  - 剩余阶段
- 风险与偏航提示
  - 信息不足
  - 预计超时
  - 预算风险
  - 结果不确定性

该栏优先绑定现有：

- `mission`
- `workflow definition / instance`
- `projection`
- `task summary`

#### 1.2 中间：执行主视图

中间区域是驾驶舱主屏，用于展示系统当前正在发生的事情。

建议承载以下信息块：

- 当前编队
  - Planner
  - Clarifier
  - Researcher
  - Generator
  - Reviewer
  - Auditor
- 当前执行态
  - 当前阶段
  - 当前节点或角色
  - 并行分支
  - 阻塞点
  - 重规划信号
- 中间产物区
  - 正在生成的文档、代码、图片、分析结果
  - 关键上下文快照
  - 推荐命令或下一步动作
- 执行事件流
  - 关键事件摘要
  - 节点状态变化
  - 最近输出

该栏优先绑定现有：

- `Mission Runtime`
- `workflow runtime engine`
- `task runtime visibility`
- `office-task-cockpit`
- `web-aigc platform mission projection`

#### 1.3 右侧：接管与证据栏

右侧用于回答两个问题：

1. 现在是否需要我介入
2. 这条路线是否可信、有据可查

建议承载以下信息块：

- 接管队列
  - 待确认路线
  - 待补上下文
  - 待审批预算
  - 待确认方向
  - 待确认最终交付
- HITL 决策面板
  - `selection`
  - `param_collection`
  - `user_input`
  - `confirm_judge`
- 证据区
  - replay 时间轴入口
  - audit 摘要入口
  - lineage 证据入口
  - 成本与权限记录

该栏优先绑定现有：

- `mission decision`
- `HITL session`
- `replay`
- `audit`
- `lineage`

### 2. 信息优先级

驾驶舱首页的信息优先级应固定如下：

1. 目标是否明确
2. 当前路线是否可理解
3. 系统当前是否在推进
4. 是否需要用户接管
5. 是否具备可信证据

因此不应让以下内容成为默认第一屏主体：

- 冗长对话记录
- 全量节点列表
- 原始 DAG 编辑器
- 低层日志流

这些信息可下钻，但不应替代主信息架构。

### 3. 与现有基座的映射

#### 3.1 与 mission-first 的映射

- `mission` -> 驾驶舱中的当前任务
- `workflow` -> 驾驶舱中的当前路线
- `task / projection` -> 驾驶舱中的阶段与执行投影
- `session` -> 驾驶舱中的交互上下文与接管上下文

#### 3.2 与 Mission Runtime 的映射

- runtime phase -> 当前阶段标签
- node run status -> 当前执行态
- waiting_input -> 接管点提示
- resume execution -> 接管完成后的继续行驶
- branch transition -> 路线分叉与选择结果
- complete / fail / retry -> 交付、异常、恢复状态

#### 3.3 与 HITL 的映射

现有人工输入与决策模型不再以“特殊节点页面”暴露，而统一映射为右侧接管区：

- `selection` -> 多选接管
- `param_collection` -> 参数补全接管
- `user_input` -> 额外上下文接管
- `confirm_judge` -> 批准 / 驳回接管

#### 3.4 与 replay / audit 的映射

- replay -> 驾驶轨迹与事件时间轴
- audit -> 决策、权限、执行和异常的审计证据
- lineage -> 数据来源、加工与输出关系

驾驶舱不重复实现 replay / audit 系统，而是提供统一入口、上下文摘要和关键证据卡片。

### 4. 页面组织原则

- 主页面默认聚焦单个任务驾驶舱
- 多任务切换通过任务列表或顶部切换器处理，不占用三栏主骨架
- 聊天记录、细粒度节点图、详细审计表格和完整回放时间轴均作为二级展开区或跳转页
- 驾驶舱必须优先服务“任务推进”，其次才是“系统解释”

## 设计约束

- 不在本 spec 中定义最终视觉稿与组件库样式
- 不在本 spec 中替换现有 replay / audit / HITL 页面，只定义驾驶舱与它们的关系
- 不在本 spec 中定义 L1-L5 自动驾驶等级
- 不在本 spec 中要求用户直接操作底层 50+ 节点
- 首版以桌面端三栏布局为主，移动端适配后续单独细化
