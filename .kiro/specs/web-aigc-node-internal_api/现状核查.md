# internal_api 节点现状核查

## 核查结论

`web-aigc-node-internal_api` 当前已经具备一条可复用的最小闭环：可以通过统一的 `InternalApiExecutor` 以 `targetId` 目录式方式调用内部能力目标，并经由 `auto-agent` / `a2a` 入口对外暴露。按当前代码与测试事实判断，4 个任务项可以继续维持完成态，而且这轮 tools-and-agents 主线增强后，最重要的新增结论是：

- internal_api 已形成“入口级治理 + 执行级 fallback”两层边界
- request metadata 已不只是透传，而是被用来解析 workflow / mission / agent / token / fallback 上下文
- audit 留痕已能区分入口钩子、治理阻断和 fallback 降级
- internal_api 结果已经稳定接入 auto_agent 统一执行面

这里需要特别说明两点：

- 这条链路已经形成“可执行的内部调用 contract”，但第一阶段落地方式是 `targetId` 目录式分发，不是完整意义上的“服务标识 + 接口路径 + 参数模板”通用协议。
- 当前已验证的是“基础报错 + 入口权限/治理 + 统一错误出口 + 最小回退策略闭环”，但还不是“按目标目录配置化、与工作流 runtime 深度联动”的高级恢复体系。

## 已完成项与依据

### 1. 定义内部接口调用结构

已完成。

依据：

- `server/tool/api/internal-api-adapter.ts`
  - 已定义统一请求结构 `InternalApiExecutionRequest`：
    - `targetId`
    - `input`
    - `context`
    - `workflowId?`
    - `stage?`
    - `metadata?`
  - 已定义统一返回结构 `InternalApiExecutionResult`：
    - `output`
    - `targetLabel`
    - `operation`
    - `response`
  - 当前元数据解析还已支持：
    - `workflowId / instanceId`
    - `missionId / taskId`
    - `agentId / requestedBy / operator`
    - `token / capabilityToken`
    - `fallback / errorFallback`
- 同文件已沉淀第一批内部目标：
  - `mission.projection.get`
  - `mission.session.get`
  - `workflow.graph_instance_snapshot`
  - `aigc_monitoring.instances`
  - `aigc_monitoring.instance_detail`
  - `aigc_monitoring.session_detail`
  - `web_aigc.risk_action_catalog`
- `server/tests/internal-api-adapter.test.ts`
  - 已验证其中多类目标的真实执行结果，包括：
    - `mission.projection.get`
    - `mission.session.get`
    - `workflow.graph_instance_snapshot`
    - `aigc_monitoring.instance_detail`
    - `web_aigc.risk_action_catalog`

限定说明：

- 当前与 `requirements.md` 中“按服务标识、接口路径、参数模板调用内部 API”的理想形态相比，主仓更接近“按 `targetId` 调用内部能力目录”的第一阶段方案。
- 但从 spec 对账角度看，调用结构本身已经被统一建模、实现并测试，因此可以认定该项已完成。

### 2. 增加权限与审计钩子

已完成。

依据：

- `server/tool/api/internal-api-adapter.ts`
  - 已支持可选注入：
    - `permissionEngine`
    - `auditLogger`
  - 当启用治理能力时，强制要求：
    - `metadata.agentId`
    - `metadata.token`
    - 并兼容 `metadata.capabilityToken`
  - 统一资源标识为：
    - `internal_api:<targetId>`
  - 会在入口执行权限检查，并结合治理策略判断是否阻断
  - 会统一记录：
    - `allowed`
    - `denied`
    - `error`
  - 已记录的审计元数据包括：
    - `targetId`
    - `workflowId`
    - `missionId`
    - `stage`
    - `inputPreview`
    - `governanceHook`
    - `fallbackUsed`
    - `fallbackStrategy`
    - `fallbackReason`
- `server/tests/internal-api-adapter.test.ts`
  - 已验证：
    - 缺失 `agentId` 时拒绝执行
    - 缺失 `token` 时拒绝执行
    - 权限拒绝时阻断执行并记录 denied 审计
    - 成功执行时记录 allowed 审计
    - fallback 不能绕过权限拒绝
    - recoverable fallback 成功后会记录 fallback 审计元数据

结论：

- 节点级统一权限、治理与审计入口已经真实落地，不只是设计文档或局部能力自带治理。

### 3. 规范输出结构

已完成。

依据：

- `server/tool/api/internal-api-adapter.ts`
  - 每个内部目标都统一返回：
    - `output`：JSON 字符串
    - `targetLabel`
    - `operation`
    - `response`：结构化对象
- `server/tool/api/auto-agent-adapter.ts`
  - 已支持 `kind: "internal_api"`
  - 会将 internal api 结果继续收敛到统一 `AutoAgentExecutionResult`
  - 会把 `request.metadata` 留在 `metadata.requestMetadata`
- `server/routes/a2a.ts`
  - `/api/a2a/auto-agent` 已接受 `kind = "internal_api"`
- `server/tests/auto-agent-adapter.test.ts`
  - 已验证 `internal_api` 结果会被 auto-agent 正确接收并转换
- `server/tests/auto-agent-routes.test.ts`
  - 已验证对外路由能够接受 `internal_api` 目标并返回结果
  - 已验证 `workflowId` 会随着对外路由请求进入统一 executor

结论：

- 输出结构不但在 adapter 内部统一，而且已经能够穿透到 auto-agent 与对外调用面。

## 未完成项与依据

### 4. 验证错误回退策略

已完成，但完成度属于“最小闭环”。

已具备的部分：

- `server/tool/api/internal-api-adapter.ts`
  - 代码里已经定义了基础错误分支：
    - `Missing required field: missionId`
    - `Missing required field: workflowId`
    - `Mission not found`
    - `Workflow not found`
    - `Internal API target not found`
    - `Missing required field: agentId`
    - `Missing required field: token`
  - 现已支持在 `metadata.fallback` 中声明最小回退策略：
    - `mode: "empty_result"`
    - `mode: "static_response"`
  - 可通过 `recoverableErrors` 指定只有命中特定错误时才允许回退
  - fallback 只发生在内部目标执行报错之后，不会覆盖入口权限检查与治理阻断
- `server/tool/api/auto-agent-adapter.ts`
  - 已有 `mapAutoAgentErrorToStatusCode()`，可将一部分错误映射为 HTTP 状态码
- `server/routes/a2a.ts`
  - `/api/a2a/auto-agent` 已具备统一错误响应出口
- `server/tests/internal-api-adapter.test.ts`
  - 现已覆盖：
    - `Mission not found` -> `empty_result` 回退
    - `Workflow not found` -> `static_response` 回退
    - 错误不在 `recoverableErrors` 中时不回退
    - 权限拒绝时即使声明 fallback 也不会绕过治理
    - 成功回退后会记录 fallback 审计元数据

当前缺口：

- 目前回退策略仍然是节点级最小实现，还没有形成按 `targetId` 维度集中配置的恢复策略目录
- 目前只验证了部分代表性错误，没有把所有 internal_api 错误分支都补成专项回退测试
- 还没有把 fallback 结果进一步接入工作流 runtime 的分支控制语义

结论：

- 当前已经可以认定“最小错误回退策略已实现并被专项验证”
- 但不能认定“完整企业级错误恢复体系已经完成”
