# 用户输入节点现状核查

## 核查范围

本次核查只对照当前主仓已经存在的共享契约、任务决策链路、审计链路与测试证据，不假设尚未落库的 `user_input` 专用实现。

## 结论摘要

当前 `web-aigc-node-user_input` 更准确的状态是：

1. Cube 已具备一套可复用的通用 HITL 等待与恢复链路，可以承接 `user_input` 类场景。
2. `waiting -> submit decision -> resume / cancel / timeout fail` 这条任务侧闭环已有实现和测试。
3. 但当前没有看到 `WorkflowRuntimeEngine` 为 `user_input` 注册显式节点适配器；也就是说，平台已经有“任务等待输入能力”，不等于“`user_input` runtime 节点已原生接入”。
4. “结构化输入”仅能通过 `metadata.formData` 兼容承接；“附件补充输入”尚未发现独立契约和专项测试。

## 已落地事实

### 1. 等待输入载荷与超时字段已经存在

`shared/mission/contracts.ts` 已提供：

- `WEB_AIGC_HITL_NODE_TYPES`
  - 包含 `user_input`
- `MissionDecision`
  - 支持 `prompt / options / allowFreeText / placeholder / payload / timeoutMs / timeoutAt`
- `MissionDecisionSubmission`
  - 支持 `optionId / freeText / submittedBy / metadata`
- `WebAigcHitlSubmissionMetadata`
  - 支持 `nodeType / sessionId / nodeId / interactionId / branchKey / formData`

这说明任务侧已经有一套可承接 `user_input` 类等待输入的通用提交契约。

### 2. 任务侧等待与恢复链路已落地

`server/tasks/mission-runtime.ts` 与 `server/tasks/mission-store.ts` 已提供：

- `MissionRuntime.waitOnMission()`
  - 把任务置为 `waiting`
- `MissionStore.markWaiting()`
  - 写入 `waitingFor / decision / timeoutAt`
- `submitMissionDecision()`
  - 校验输入、生成 `resolved decision`
- `MissionRuntime.resumeMissionFromDecision()`
  - 通过 `MissionStore.resolveWaiting()` 把任务恢复为 `running`
- `MissionStore.resolveWaiting()`
  - 清理等待态，写入 `decisionHistory`，追加恢复进度事件

这条链路已经满足“先等待、后提交、再恢复”的通用要求。

### 3. 决策接口、历史与审计链路已落地

`server/routes/tasks.ts` 已提供：

- `POST /api/tasks/:id/decision`
- `GET /api/tasks/:id/decisions`

`server/audit/audit-hooks.ts` 已把 `emitDecisionSubmitted()` 桥接到：

- `human.decision_submitted`
- `DECISION_MADE` 审计事件

因此，输入提交后的历史记录和审计记录都已有真实落点，而不只是前端瞬时状态。

### 4. 人工提交、人工中断与超时场景已有测试证据

当前可直接引用的测试证据包括：

- `server/tests/hitl-decision.test.ts`
  - 验证 `POST /api/tasks/:id/decision` 能提交并恢复任务
  - 验证 `submittedBy / sessionId / interactionId / branchKey` 等 metadata 会进入 `decisionHistory`
  - 验证会写入 `human.decision_submitted` 审计记录
  - 验证超时等待输入会被拒绝提交
- `server/tests/mission-store.test.ts`
  - 验证 `expireWaiting()` 会把超时任务标记为失败
  - 验证会记录 `User input timed out.` 失败事件
- `server/tests/mission-cancel.test.ts`
  - 验证等待中的 mission 可以被取消
  - 验证取消后会清理 `waitingFor` 与 `decision`
  - 验证等待态 `user_input` 不会残留未完成的输入检查点

就“人工提交、人工中断与超时场景是否已验证”这一项而言，当前证据是成立的。

## 尚未完整落地或未见证据的部分

### 1. 未看到 `user_input` 显式 runtime 节点适配器

`server/core/workflow-runtime-engine.ts` 当前内置注册的是：

- `selection`
- `confirm_judge`
- 以及 `root / agent_task / plan / review / audit / summary / end`

但没有看到 `user_input` 的显式 adapter 注册。

这意味着：

- 任务侧等待输入能力已经有了
- 但 `user_input` 节点并未像 `selection / confirm_judge` 一样，明确接入 workflow runtime 的 `wait / resume / branch` 执行器

### 2. 结构化输入支持仍是兼容承接，不是独立表单契约

虽然 `WebAigcHitlSubmissionMetadata.formData` 已存在，但当前未看到：

- `MissionDecision` 直接承载字段级 schema
- `WebAigcHitlFieldDefinition` 被 `MissionDecision` 或 `tasks` 路由正式消费
- `user_input` 专用表单 schema 的测试

因此，当前更准确的说法是：

- 支持“把结构化输入塞进 metadata.formData”
- 尚不等于“平台已经具备 `user_input` 独立字段表单契约”

### 3. 附件补充输入尚未看到独立提交字段

当前未发现：

- `MissionDecisionSubmission` 中有附件字段
- `user_input` 节点附件补充输入的后端提交契约
- `user_input` 附件输入的专项测试

所以 `requirements.md` 中“附件补充输入”这一层，目前没有足够证据证明已经完整落地。

## 对 tasks.md 的判断

| task | 当前判断 | 结论 |
| --- | --- | --- |
| `定义等待输入节点载荷` | 保持已完成 | 通用等待输入载荷已经存在，主要落在 `MissionDecision` 与 `metadata.formData / timeoutAt`。 |
| `复用任务输入与决策接口` | 保持已完成 | `waitOnMission()` 与 `POST /api/tasks/:id/decision` 已构成可复用任务接口。 |
| `写入输入提交与恢复事件` | 保持已完成 | `decisionHistory`、恢复进度事件、Socket 决策广播和审计记录都已存在。 |
| `验证人工中断与超时场景` | 保持已完成 | `hitl-decision`、`mission-store` 与 `mission-cancel` 测试已覆盖人工提交恢复、等待态取消清理与超时失败。 |

## 风险与后续建议

如果后续要把 `user_input` 从“通用 HITL 兼容承接”推进到“节点原生落地”，优先建议补这 3 件事：

1. 在 `WorkflowRuntimeEngine` 中为 `user_input` 注册显式 adapter，并提供 `wait / resume` 执行入口。
2. 把 `WebAigcHitlFieldDefinition` 真正接入 `MissionDecision` 或 checkpoint schema，形成字段级结构化输入契约。
3. 为附件补充输入增加独立提交字段与测试，避免长期只停留在文字或 metadata 兼容层。
