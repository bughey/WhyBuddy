基于 2026-04-23 当前主仓代码事实，`web-aigc-node-open_dashboard` 现在可以确认完成 4 项中的 4 项：

- 已完成：`定义看板打开目标结构`
- 已完成：`增加参数透传设计`
- 已完成：`增加权限校验`
- 已完成：`验证前端联动`

本次实现没有去补真实前端页面，而是先把看板打开节点需要的最小后端闭环补齐：节点执行后能返回前端可消费的 dashboard `target` 描述、能透传上下文参数、能做权限校验、能通过独立路由供宿主前端读取目标描述。

## 当前最小闭环是什么

### 1. 已定义可打开 dashboard 的目标结构

当前 `open_dashboard` 已通过 `server/routes/node-adapters/open-dashboard-node-adapter.ts` 接入统一节点执行结构。

输出中的 `target` 当前包含：

- `kind`
- `dashboardId`
- `route`
- `title`
- `description`
- `apiHref`
- `uiHref`
- `context`

这意味着前端即使还没有真实看板页面，也已经可以根据该结构决定：

- 跳哪个 UI 路由
- 请求哪个后端目标描述接口
- 透传哪些上下文参数

### 2. 已支持参数透传

当前节点输入支持：

- `dashboardId`
- `route`
- `title`
- `description`
- `context`

其中 `context` 会被保留到：

- 节点输出 `output.context`
- `target.context`

因此当前已具备“把筛选条件、业务上下文、来源参数一起交给前端消费”的最小能力。

### 3. 已补最小权限校验

当前节点适配器已支持在接入全局 `PermissionCheckEngine` 时执行最小权限门禁。

当前权限口径是：

- 资源类型：`api`
- 动作：`call`
- 资源：`GET /api/open-dashboard/targets/:dashboardId`

启用权限引擎后：

- 缺少 `agentId` / `token` 会直接报错
- 权限拒绝会返回 `denied`
- 路由层会映射为 `403`

这满足当前 spec 所要求的最小权限闭环。

### 4. 已具备前端可消费的目标描述接口

当前已新增：

- `POST /api/open-dashboard/nodes/execute`
- `GET /api/open-dashboard/targets/:dashboardId`

其中：

- `POST` 负责执行节点、返回完整 `target`
- `GET` 负责在宿主前端需要时读取 dashboard 目标描述

因此当前“前端联动”按最小口径已经成立：

- 后端已返回前端可消费的打开目标
- 不要求当前仓库中已经存在真实 dashboard 页面

## 仍需保留的边界

虽然当前已形成最小闭环，但仍应明确以下边界：

- 当前没有真实前端 dashboard 页面实现。
- 当前权限口径是最小 `api:call` 校验，不是完整页面级权限模型。
- 当前没有引入多租户 dashboard registry、动态菜单同步、可视化配置中心等更重能力。

后续如果要继续增强，可以考虑：

- 接入真实 dashboard registry
- 将 `dashboardId` 与宿主前端路由表做强绑定
- 增加更细粒度的页面访问控制与审计事件
