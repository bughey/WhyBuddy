# 设计文档：任务自动驾驶平台定位

## 设计概述

本 spec 负责定义“任务自动驾驶平台”的产品定位，不直接承载具体节点实现、运行时接口或页面细节，而是为后续 `core concepts`、`L1-L5`、`cockpit IA`、`route planner` 等 spec 提供统一叙事基础。

该定位层的目标不是替换现有技术实现命名，而是建立一层稳定的产品语言，把现有 `mission-first` 能力重新组织为用户更容易理解和接受的系统表达。

## 核心设计

### 1. 平台定义

对外统一定义为：

> `Cube Pets Office` 是一个面向复杂任务的任务自动驾驶平台。用户输入目标，系统自动理解目的地、规划路线、组织 Agent 编队、在执行中持续澄清与校正，并把任务送达结果。

对内统一理解为：

- 产品层：`destination / route / drive state / fleet / takeover`
- 运行层：`mission / workflow / runtime / review / audit / replay`

产品层负责用户理解，运行层负责工程承接。

### 2. 与相邻产品类型的差异

#### 2.1 与 Chat Playground 的差异

Chat Playground 的核心对象是“对话回合”，默认路径是“提问 - 回答 - 追问”。

任务自动驾驶平台的核心对象是“目标送达”，默认路径是：

1. 输入目标
2. 识别目的地与约束
3. 生成路线
4. 自动组织执行
5. 在关键点请求接管
6. 输出交付结果

因此本平台不以聊天框轮次为中心，而以任务推进状态为中心。

#### 2.2 与 Workflow Builder 的差异

Workflow Builder 的核心动作是“搭图”，强调人工配置节点、边、条件和分支。

任务自动驾驶平台的核心动作是“发车”，强调系统根据目标自动给出推荐路线、自动分配能力包、自动治理执行过程。用户可以查看和调整路线，但不要求用户从零搭建流程图。

#### 2.3 与 Agent Platform 的差异

Agent Platform 更强调能力市场、Agent 列表、工具列表、模型编排和插件生态。

任务自动驾驶平台将这些能力收敛为完成任务的执行编队，不以“有哪些 Agent”作为主要卖点，而以“是否能稳定送达结果”作为主要价值。

### 3. 与 mission-first 的关系

现有 `mission-first` 定义保留，作为底层工程哲学：

- 用户不是执行零散动作，而是在推进一个 mission
- 系统不是返回单个答案，而是在运行一条任务链

“任务自动驾驶平台”是在此基础上的产品升级：

- `mission` 对外表达为 `destination`
- `workflow` 对外表达为 `route`
- `runtime state` 对外表达为 `drive state`
- `human approval / clarification` 对外表达为 `takeover point`

这样可以在不推翻既有实现的情况下，升级用户认知与产品包装。

### 4. 平台边界

当前平台明确覆盖：

- 复杂任务目标接收与目标摘要
- 路线推荐与阶段拆解
- Agent / 节点编队与执行分工
- 过程中的澄清、审批、接管与重规划
- 结果交付、证据展示、回放与审计

当前平台明确不承诺：

- 开放域 L5 全自动执行
- 不受预算、权限、上下文限制的无限自治
- 对所有任务一次成功且无需校验
- 让用户直接管理全部底层节点细节

### 5. 对外表达约束

后续 README、官网、介绍材料应统一遵守以下表达约束：

- 可以说“不是对话式 AI，而是任务自动驾驶”
- 可以说“输入目标，系统规划路线并组织执行”
- 可以说“必要时由用户接管关键决策”
- 不应说“完全替代人工”
- 不应说“自动完成任何复杂任务”
- 不应把底层 50+ 节点直接作为产品主入口

## 设计约束

- 不在本 spec 中定义具体页面布局，页面布局由 `autopilot-cockpit-information-architecture` 负责
- 不在本 spec 中定义具体状态机与接口字段，状态机由 `drive-state-and-replan-state-machine` 负责
- 不在本 spec 中定义自动化等级细则，等级定义由 `task-autopilot-levels-l1-to-l5` 负责
- 不在本 spec 中改变现有底层命名，仅定义产品层映射方式
