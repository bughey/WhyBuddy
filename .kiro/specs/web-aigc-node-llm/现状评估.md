# web-aigc-node-llm 现状评估

## 定位说明

本文作为 `现状核查.md` 的补充材料，重点收口“测试链路与验收边界”。

- `现状核查.md` 负责给出任务项是否可勾的主结论。
- 本文负责说明这些结论主要建立在哪些运行路径与测试证据之上，以及哪些场景不应被误判为已经完全收口。

## 主结论补充

截至 2026-04-22，`web-aigc-node-llm` 保持 `4 / 4` 已完成是成立的，但成立前提是：

- 判定依据以服务端主链路为主。
- 结构化输出、provider fallback、telemetry、cost 都已有实现与测试支撑。
- 浏览器直连模式仍属于边界保留项，不纳入“与服务端同等完成度”的表述。

## 测试链路补充

### 1. 结构化输出链路

以下测试证明 `jsonMode` 与结构化输出不是停留在接口定义，而是已经进入主调用路径：

- `server/tests/agent-vision-events.test.ts`
  - 覆盖 `invokeJson()` 路径。
- `server/tests/nl-command/decision-support.test.ts`
  - 断言调用参数包含 `jsonMode: true`。
- `server/tests/nl-command/command-analyzer.test.ts`
  - 断言调用参数包含 `jsonMode: true`。
- `server/tests/llm-client.provider-fallback.test.ts`
  - 证明 `callLLMJson()` 在 provider fallback 场景下仍可返回结构化结果。

### 2. 指标写入与读取链路

以下测试证明成本、延迟与 token 指标已经落到可观测链路：

- `server/tests/telemetry.test.ts`
  - 验证 LLM 调用记录可进入 telemetry snapshot。
- `server/tests/cost-tracker.test.ts`
  - 验证 `recordCall()` 的累计、聚合与阈值逻辑。
- `server/tests/cost-api.test.ts`
  - 验证成本接口可读取累计 token / cost / call 指标。

## 仍需保留的边界

以下事项建议继续保留在“未完全收口”口径中：

- 浏览器直连模式下的 token / cost 精度仍偏弱。
- 端侧观测结果与服务端观测结果不能简单视为完全等价。
- 若未来把“浏览器直连精确计量”纳入本 spec 的严格验收，需要补单独实现与测试。

## 文档使用建议

- 查看任务是否可勾，以 `tasks.md` 与 `现状核查.md` 为准。
- 查看为什么能勾、证据主要落在哪些测试上，可读本文。
- 若后续主仓实现发生变化，优先同步更新 `现状核查.md`，再视情况补充本文。
