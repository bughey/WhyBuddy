# 需求文档：LLM 节点

## 目标

定义 `llm` 节点作为通用大模型调用节点的能力边界，使其能够被多个工作流节点和运行时链路复用，而不是只服务某个单页功能。

## 当前主仓边界（2026-04-22）

- 服务端主链路已经具备统一的 LLM 输入输出契约、provider 抽象、多提供商调用与 fallback 能力。
- 已支持模型、温度、`maxTokens`、`jsonMode` 等核心参数。
- 已支持结构化输出路径，并通过 telemetry / cost 链路写入延迟、token、成本指标。
- 当前已完成口径主要基于服务端主链路与共享 runtime 抽象。
- 浏览器直连模式下的 token / cost 统计精度仍偏弱，不应与服务端主链路视为同等完成度。

## 需求

### 需求 1：通用生成

系统应支持以提示词、变量、上下文为输入进行通用大模型调用，并输出标准化响应结果。

### 需求 2：参数配置

应支持模型、温度、最大输出长度、结构化输出开关等基础配置，以满足不同节点的复用需求。

### 需求 3：结果标准化与观测

输出应支持文本、JSON 或结构化结果，并纳入延迟、token、成本等观测链路。

## 暂不扩大验收口径的事项

- 浏览器直连模式下与服务端完全等价的 token / cost 精确采集。
- 所有端侧与服务端调用路径在观测精度上的完全一致性。
