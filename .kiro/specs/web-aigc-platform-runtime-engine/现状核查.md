# web-aigc-platform-runtime-engine 现状核查

## 结论

截至 2026-04-23，`web-aigc-platform-runtime-engine` 这组任务已经可以按“最小真实运行时闭环”认定为完成。

本轮把此前唯一未收口的“治理策略闭环”补到了可验收状态，关键变化不是再增加文档口径，而是把原本分散在节点级的重试/升级逻辑，提升成了实例级统一治理策略层：

- 自动重试和显式重试现在都会受同一套实例级预算约束
- 自动重试的默认退避现在可以由实例级治理策略统一下发
- 预算耗尽时会留下结构化阻断证据，而不是静默失败
- 阻断后可按治理策略转入自动升级
- 控制面事件会携带治理快照进入 replay / audit

## 本轮新增并已落地的治理能力

### 1. 实例级统一治理策略层

`server/core/workflow-runtime-engine.ts` 现已支持最小治理配置：

- `maxAutomaticRetries`
- `maxManualRetries`
- `maxTotalRetries`
- `retryDelayMs`
- `escalateOnRetryBlocked`

这套策略可以通过两种方式进入运行时：

- `definition.metadata.runtimeGovernance`
- `POST /api/workflows/:id/runtime/run` 的 `runtimeGovernance` 请求体

### 2. 治理预算真正影响运行时行为

当前不再只是“节点自己看自己的 `retryBudget`”。

现在运行时会同时考虑：

- 节点级策略：
  - `retryBudget`
  - `retryDelayMs`
  - `autoEscalateOnFailure`
  - `escalateOnRetryExhausted`
- 实例级统一治理策略：
  - 自动重试预算
  - 手动重试预算
  - 总重试预算
  - 默认退避
  - 阻断后是否升级

也就是说，节点级策略仍然保留，但最终会被实例级治理预算进一步约束。

### 3. 治理状态与阻断证据可持久化

运行时变量里现在会留下：

- `runtimeGovernancePolicy`
- `runtimeGovernanceState`
- `runtimeRetryBlocked`

因此当前已具备最小可审计的治理证据，而不是只有“重试失败/升级成功”这种最终表象。

### 4. 治理元数据进入控制面事件

当前以下控制面事件已可携带治理快照：

- `instance.retry_requested`
- `instance.escalated`
- `instance.terminated`

`server/core/web-aigc-runtime-observability.ts` 会把这些事件继续镜像到 replay / audit，因此治理状态已进入平台可观测面。

## 本轮新增测试证据

### 运行时内核测试

`server/tests/workflow-runtime-engine.test.ts` 已新增：

- 实例级自动重试预算跨节点生效
  - 第一个节点消耗掉唯一一次自动重试额度后，后续节点即使自身仍有 `retryBudget`，也会被实例级治理预算阻断
- 实例级手动重试预算耗尽后阻断显式 `retry()`
  - 显式控制面不再绕过治理预算

### 路由测试

`server/tests/workflows-routes.test.ts` 已新增：

- `/runtime/run` 会把 `runtimeGovernance` 注入到 runtime definition 的 `metadata.runtimeGovernance`
- `/runtime/retry` 在治理预算阻断时稳定返回 `409`
- 路由层会直接透出 `Runtime retry blocked by governance policy: ...` 错误文案，便于主线控制面据此做治理态提示

### 观测测试

`server/tests/web-aigc-runtime-observability.test.ts` 已新增：

- `instance.retry_requested / instance.escalated` 的治理快照会进入 replay / audit 元数据

## 仍然存在的边界

这次补齐的是“最小真实治理闭环”，不是完整治理中台。当前仍未实现：

- 独立后台重试队列
- 独立指数退避服务
- 跨实例统一治理中台
- runtime 事件直写 lineage

但这些缺口已经不再阻塞当前 spec 的最小验收，因为：

- 控制面已存在
- 自动策略已存在
- 实例级统一预算/退避/升级控制已存在
- 路由、测试、观测都已能看到这层治理

## 当前最准确的判定

`web-aigc-platform-runtime-engine` 现在可以按以下口径收口：

- 图执行闭环：已成立
- HITL 等待/恢复闭环：已成立
- 内置节点最小 runtime 能力：已成立
- 控制面与最小统一治理闭环：已成立
- replay / audit / 公共事件面镜像：已成立

因此，当前顶层任务可按完成态维护。  
后续若要继续增强，应把目标从“补齐最小治理闭环”升级为“建设独立治理中台/策略服务”。
