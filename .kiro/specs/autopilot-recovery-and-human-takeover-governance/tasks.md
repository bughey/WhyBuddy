# 任务清单：任务自动驾驶恢复机制与人工接管治理

## 任务

- [ ] 1. 定义恢复与接管治理契约层
  - [ ] 1.1 定义 `DeviationEvent` 契约，覆盖偏航、失败、阻塞与恢复耗尽分类。
  - [ ] 1.2 定义 `DeviationCategory`、触发来源和严重级别枚举。
  - [ ] 1.3 定义 `RecoveryStrategy`、`RecoveryStrategyType` 与 `RecoveryImpact` 契约。
  - [ ] 1.4 定义 `RecoveryAttempt` 契约，记录恢复尝试、治理判定和执行结果。
  - [ ] 1.5 定义恢复场景下的 `RecoveryDecisionPoint` 与选项结构。
  - [ ] 1.6 定义恢复相关事件与现有 `Drive State`、`Takeover Point` 的映射字段。

- [ ] 2. 梳理偏航检测信号来源
  - [ ] 2.1 梳理 runtime 侧失败、超时、重试耗尽、执行器不可用等真实信号来源。
  - [ ] 2.2 梳理 `review / verify / revise` 侧可触发恢复的真实信号来源。
  - [ ] 2.3 梳理 `audit / runtime governance` 侧预算、权限、风险、外部副作用命中信号。
  - [ ] 2.4 梳理用户手动反馈、人工改线、目标变更等人工触发信号。
  - [ ] 2.5 定义强触发、弱触发和多信号叠加升级规则。

- [ ] 3. 设计偏航检测与恢复分类器
  - [ ] 3.1 设计 `DeviationDetector` 的输入、输出与事件投影方式。
  - [ ] 3.2 定义 `goal_deviation`、`route_deviation`、`quality_deviation` 的识别规则。
  - [ ] 3.3 定义 `governance_deviation`、`dependency_failure`、`state_block` 的识别规则。
  - [ ] 3.4 定义 `recovery_exhausted` 的判定条件与升级规则。
  - [ ] 3.5 明确检测结果如何映射到 `blocked / takeover-required / replanning`。

- [ ] 4. 设计恢复策略规划器
  - [ ] 4.1 设计 `RecoveryCoordinator` 的总体流程。
  - [ ] 4.2 定义从局部重试到异常升级的恢复层级顺序。
  - [ ] 4.3 定义不同异常类型的首选、次选和禁止策略矩阵。
  - [ ] 4.4 区分普通 `retry`、`revise`、`replan` 与正式恢复动作的边界。
  - [ ] 4.5 定义恢复动作的退出条件、放弃条件与升级条件。

- [ ] 5. 设计自动恢复机制
  - [ ] 5.1 定义节点级重试、替代执行器切换的触发与上限。
  - [ ] 5.2 定义从快照或稳定检查点恢复的最小能力。
  - [ ] 5.3 定义跳过非关键步骤继续执行的准入条件。
  - [ ] 5.4 定义局部自动恢复失败后进入下一层恢复的规则。
  - [ ] 5.5 定义自动恢复记录如何进入 replay / audit / evidence。

- [ ] 6. 设计降级执行机制
  - [ ] 6.1 定义模型降级、路线降级、权限降级和范围降级的能力边界。
  - [ ] 6.2 定义降级前的治理校验项。
  - [ ] 6.3 定义降级对时间、成本、质量、风险和自动化程度的影响表达。
  - [ ] 6.4 定义禁止静默降级的高风险场景。
  - [ ] 6.5 定义降级执行与现有 cost governance、permission governance、runtime governance 的对接方式。

- [ ] 7. 设计人工接手与人工确认机制
  - [ ] 7.1 明确“人工确认后继续”和“人工直接接手”的语义区别。
  - [ ] 7.2 定义按 `step / stage / route / mission` 的接手范围模型。
  - [ ] 7.3 设计恢复场景下的接手选项，如继续、降级、改线、升级、终止。
  - [ ] 7.4 定义哪些恢复动作必须要求评论或责任确认。
  - [ ] 7.5 定义人工接手完成后如何回到自动链路继续。

- [ ] 8. 设计与现有 HITL / decision / wait-resume 的兼容层
  - [ ] 8.1 将恢复决策点映射到现有 `Takeover Point` 类型。
  - [ ] 8.2 将恢复选项映射到现有 `MissionDecision` 类型与 payload。
  - [ ] 8.3 复用 `submitMissionDecision()` 的幂等提交流程。
  - [ ] 8.4 定义恢复后继续如何映射到 `resume()`、orchestrator 继续或 `replan`。
  - [ ] 8.5 定义异常升级如何映射到 `escalate()`。

- [ ] 9. 设计与 review / audit / revise / verify 的闭环兼容
  - [ ] 9.1 定义 `review` 失败进入恢复或 revise 的策略。
  - [ ] 9.2 定义 `verify` 失败进入恢复、接管或终止的策略。
  - [ ] 9.3 定义 `revise` 成功后返回 `reviewing` 或 `executing` 的路径。
  - [ ] 9.4 定义 `audit` 命中治理问题后进入接管或升级的规则。
  - [ ] 9.5 统一质量失败与运行失败在时间线中的表达方式。

- [ ] 10. 设计与 runtime governance 的治理对接
  - [ ] 10.1 定义恢复动作需要检查的治理维度：重试预算、成本、权限、风险、外部副作用、自动化等级。
  - [ ] 10.2 设计 `allowed / denied / needs_takeover / needs_escalation` 的治理判定结果。
  - [ ] 10.3 定义治理命中后的默认控制面：阻断、建议接管、必须接管或升级。
  - [ ] 10.4 定义恢复动作对自动驾驶等级降级的影响。
  - [ ] 10.5 定义治理判断与恢复动作如何进入审计链。

- [ ] 11. 设计异常升级机制
  - [ ] 11.1 定义自动恢复耗尽、风险过高、权限越界、外部副作用等升级触发条件。
  - [ ] 11.2 定义升级前必须保留的上下文、证据与人工评论。
  - [ ] 11.3 定义升级后任务是冻结、等待更高权限接手还是终止。
  - [ ] 11.4 定义异常升级与现有人工值守、治理审批或终止控制面的关系。
  - [ ] 11.5 定义升级失败或无人接手时的兜底策略。

- [ ] 12. 设计恢复状态机与 Drive State 映射
  - [ ] 12.1 定义偏航检测、恢复执行、等待接手、重新规划和恢复完成的高层状态流。
  - [ ] 12.2 明确 `blocked`、`takeover-required`、`replanning` 的边界。
  - [ ] 12.3 明确恢复后继续回到 `executing`、`reviewing` 或 `delivered` 的条件。
  - [ ] 12.4 设计恢复状态如何被服务端事件或投影层重建。
  - [ ] 12.5 与现有 `Drive State` spec 做术语和迁移条件对齐。

- [ ] 13. 设计恢复记录、回放与审计
  - [ ] 13.1 定义 `RecoveryAttemptLedger` 的最小字段和保留策略。
  - [ ] 13.2 设计 replay 时间线中偏航、恢复、接手、升级事件的展示语义。
  - [ ] 13.3 设计 audit 中高风险恢复、降级、人工批准与终止事件的记录方式。
  - [ ] 13.4 定义恢复事件与 Route、Mission、Workflow、Runtime Event 的关联键。
  - [ ] 13.5 定义默认自动恢复为何被允许的证据表达。

- [ ] 14. 设计前端展示与驾驶舱接入
  - [ ] 14.1 设计任务详情中恢复状态、恢复历史与当前接手点的展示区域。
  - [ ] 14.2 设计驾驶舱中偏航、恢复中、等待接手、升级中的状态提示。
  - [ ] 14.3 设计恢复决策面板中的推荐动作、已尝试动作与影响说明。
  - [ ] 14.4 设计降级执行的差异展示与风险提示。
  - [ ] 14.5 设计“不接手会发生什么”的提示表达。

- [ ] 15. 设计服务端投影与持久化
  - [ ] 15.1 定义恢复事件和恢复尝试如何进入 Mission 快照或等价持久化层。
  - [ ] 15.2 定义服务重启后恢复中的任务如何重新 attach 恢复上下文。
  - [ ] 15.3 定义恢复记录如何与现有 `decisionHistory`、任务历史和 audit 关联。
  - [ ] 15.4 定义恢复中的任务在重连、刷新、重启后的最小继续保证。
  - [ ] 15.5 定义历史恢复记录的查询接口或投影接口。

- [ ] 16. 补齐测试与验证策略
  - [ ] 16.1 设计偏航检测、恢复层级选择与治理判定的单元测试。
  - [ ] 16.2 设计自动恢复、降级执行、人工接手、异常升级的集成测试。
  - [ ] 16.3 设计 `review / audit / revise / verify` 进入恢复闭环的回归测试。
  - [ ] 16.4 设计 replay / audit 能重建恢复时间线的验证用例。
  - [ ] 16.5 设计服务重启、页面刷新、socket 重连后的恢复继续验证。

- [ ] 17. 制定灰度与落地计划
  - [ ] 17.1 制定第一阶段只做语义与事件投影的落地范围。
  - [ ] 17.2 制定第二阶段接入接管面板与决策桥接的落地范围。
  - [ ] 17.3 制定第三阶段接入驾驶舱、replay、audit 的落地范围。
  - [ ] 17.4 制定自动恢复阈值、降级阈值和升级阈值的灰度策略。
  - [ ] 17.5 制定回滚方案，确保新治理逻辑不会破坏现有任务链路。
