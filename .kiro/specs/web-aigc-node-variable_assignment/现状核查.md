# 变量赋值节点现状核查

## 核查结论

截至 2026-04-23，`web-aigc-node-variable_assignment` 的当前状态不再只是“控制流快照侧存在定义”，而是已经形成了“控制流快照层 + runtime 执行层 + observability 镜像层”的三段式最小闭环。

按当前主仓事实，这份 spec 的 4 个任务项都可以视为已完成，但需要带上两个口径边界：

1. runtime 主变量平面仍以 `instance.variables` 为中心，`local / temp` 目前主要体现为附加作用域快照，而不是完全隔离的独立变量容器。
2. `variable.assigned` 事件已经由 `server/core/web-aigc-runtime-observability.ts` 镜像到 replay / audit，并补齐了统一的资源与动作 metadata；当前剩余边界主要在 lineage 扩展、批量赋值聚合与更细粒度脱敏，而不是 replay / audit 是否接通。

## 一、控制流快照层已经成立

当前 `server/core/web-aigc-controlflow.ts` 已经内建 `variable_assignment` 节点的控制流快照推导逻辑，事实包括：

- `ControlFlowNodeType` 已包含 `variable_assignment`
- 节点配置已支持：
  - `target`
  - `scope`
  - `source`
  - `value`
  - `expression`
- 推导执行时会按 `global / local / temp` 作用域写入上下文
- 会把变量写入结果沉淀到 `variableChanges`
- graph projection 会输出赋值摘要，例如 `global.reviewResult = true`

对应证据：

- `server/core/web-aigc-controlflow.ts`
- `server/tests/workflow-graph-projection.test.ts`

这说明 `variable_assignment` 在“控制流快照 / replay projection”这层早就不是空白节点。

## 二、runtime 执行层已经形成最小内置闭环

当前 `server/core/workflow-runtime-engine.ts` 已注册 `VariableAssignmentWorkflowNodeAdapter`，并在 `registerBuiltInAdapters()` 中作为内置适配器接入。runtime 事实包括：

- 节点类型：`variable_assignment`
- 最小执行输入已支持：
  - `target`
  - `scope`
  - `source`
  - `expression`
  - `value`
- 运行时会解析：
  - 直接常量赋值
  - `$.path` 取值
  - 简单比较表达式，如 `score >= 90`
- 赋值结果会写回 `instance.variables`
- 同时会补充变量作用域与变更痕迹：
  - `runtimeVariableScopes`
  - `runtimeVariableChanges`
  - `runtimeVariableLastChange`
  - `lastAssignedVariable`
  - `lastAssignedScope`
  - `lastAssignedValue`
- runtime 还会发出 `variable.assigned` 事件

这代表 `variable_assignment` 已经从“图快照承接”推进到“runtime 原生执行”。

对应证据：

- `server/core/workflow-runtime-engine.ts`
- `server/tests/workflow-runtime-engine.test.ts`

## 三、与条件节点的联动已经有测试覆盖

当前主仓已经存在专门测试，验证 `variable_assignment` 产出的变量能够被后续 `condition` 节点直接消费：

- `executes variable_assignment nodes through the runtime built-in adapter`
- `lets variable_assignment feed downstream condition nodes in runtime`

这两条测试共同证明了两件事：

1. `variable_assignment` 自身的 runtime 赋值、作用域快照、变更记录是成立的。
2. `variable_assignment -> condition` 的下游联动是成立的。

这意味着“变量赋值只是写了个中间结果，但后续节点用不上”的风险已经被主仓测试显式压住。

## 四、当前完成口径

### 任务 1：定义变量作用域模型

可视为已完成。

当前口径：

- 控制流快照层与 runtime 层都已经显式保留 `global / local / temp`
- 但 runtime 的 `local / temp` 仍然依赖 `runtimeVariableScopes` 这层附加快照，而不是完全隔离的独立运行时命名空间

### 任务 2：定义赋值与表达式执行结构

可视为已完成。

当前已覆盖：

- 常量赋值
- `source` 取值
- `expression` 最小表达式求值

但当前表达式能力仍然是最小子集，不应夸大为完整表达式引擎。

### 任务 3：写入变量变更事件

可视为已完成，且当前已经进入 observability 镜像层。

当前已成立：

- 控制流快照层有 `variableChanges`
- runtime 状态层有 `runtimeVariableChanges / runtimeVariableLastChange`
- runtime 事件面有 `variable.assigned`
- replay / audit 已镜像 `variable.assigned`
- replay / audit 两侧会共享一致的赋值镜像 metadata：
  - `actionId = variable.assign`
  - `resourceType = workflow-variable`
  - `resourceId = metadata.target`
  - `assignmentScope / assignmentTarget / assignmentChanged`

因此这项现在应理解为“节点级变量变更记录、runtime 事件、replay / audit 镜像”三层都已到位；尚未完全闭环的是 lineage 与更复杂治理增强，而不是最小审计镜像本身。

### 任务 4：验证与条件节点联动

可视为已完成。

因为当前主仓已经有实际 runtime 测试证明：

- `variable_assignment` 先写入变量
- `condition` 再读取该变量
- 分支边会按条件结果继续推进

## 五、仍需关注的边界

当前还不能夸大的点主要有 3 个：

1. `local / temp` 作用域在 runtime 中还不是完全隔离的主变量平面。
2. 表达式求值仍属于最小能力，没有扩展到复杂语法、批量赋值、嵌套路径写回。
3. `variable.assigned` 虽然已经进入 replay / audit，但还没有继续扩展到 lineage 直写、批量赋值聚合审计和字段级脱敏策略。

## 六、结论

如果按“节点是否已经在主仓具备最小可运行闭环”的标准判断，`web-aigc-node-variable_assignment` 已经可以视为已收口：

- 控制流快照层：已成立
- runtime 原生执行层：已成立
- replay / audit 镜像层：已成立
- 与条件节点联动：已成立
- 中文 tasks 完成状态：可以保持全勾选

但如果后续要继续把它从“最小闭环”推进到“治理闭环”，优先建议补的是：

1. 明确 `local / temp` 的真正运行时隔离语义
2. 补变量事件的 lineage 直写与跨镜像关联索引
3. 再补更复杂表达式、批量赋值与嵌套变量路径写回
