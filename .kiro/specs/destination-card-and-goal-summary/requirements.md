# 需求文档：目的地卡片与目标摘要

## 目标

定义任务自动驾驶界面中的“目的地卡片与目标摘要层”，将用户目标、成功标准、约束、预期交付物、置信度与缺失信息整理为统一的高层展示对象。

本 spec 关注的是“用户在进入任务后，系统如何把目的地解释清楚”，而不是底层路线规划或执行细节本身。

## 需求

### 需求 1：系统应提供统一的目的地卡片视图

系统应将用户输入或已存在任务抽象为一张高层目的地卡片，并统一展示以下内容：

- 目标概述
- 成功标准
- 关键约束
- 预期交付物
- 当前置信度
- 缺失信息

该卡片应成为任务自动驾驶左侧“目标与路线”区域的首要信息锚点。

### 需求 2：系统应支持目标概述的稳定摘要

系统应将原始输入、历史上下文或 mission 描述整理为稳定的目标概述，而不是直接回显原始 prompt。

目标概述应满足：

- 能一句话说明“要送达什么结果”
- 能区分目标本体与执行手段
- 在执行过程中保持相对稳定，仅在用户修改目标或重解析后更新

### 需求 3：系统应展示成功标准与约束

系统应将“什么算完成”与“在什么条件下完成”显式展示出来，包括但不限于：

- 结果质量要求
- 时间要求
- 成本或预算约束
- 风格、格式、范围限制
- 权限、合规或安全限制

如果系统无法完整识别成功标准或约束，应在卡片中显示为待补充信息，而不是默认为空。

### 需求 4：系统应展示预期交付物

系统应明确展示系统准备交付的产物类型，用于帮助用户判断目标是否被正确理解。

预期交付物可包括：

- 文档
- 方案
- 表格
- 架构图
- 图像素材
- 代码与变更
- 任务结论与建议

预期交付物应可与现有 mission / workflow 的输出定义形成映射。

### 需求 5：系统应展示当前置信度

系统应提供与目的地理解相关的当前置信度展示，用于说明系统对目标理解是否稳定。

置信度展示应满足：

- 可用分层等级或简化分值表示
- 能说明高、中、低的基本含义
- 能与缺失信息、歧义和接管建议关联

### 需求 6：系统应显式展示缺失信息

当系统判断当前目的地仍缺少关键上下文时，应在卡片中显式展示缺失信息，而不是将其隐藏在日志或后续澄清中。

缺失信息应支持以下表达：

- 缺什么
- 为什么缺
- 缺失会影响什么
- 是否阻塞继续执行

### 需求 7：系统应与 mission-first / workflow / runtime 兼容

系统应支持将目的地卡片中的字段映射到现有工程对象：

- 目标概述映射到 `mission` 的高层描述
- 成功标准与约束映射到 `mission metadata`、runtime context 或 workflow config
- 预期交付物映射到 workflow 输出定义与任务产物
- 当前置信度映射到解析阶段或运行时评估结果
- 缺失信息映射到 clarification / HITL / input request

系统不应要求现有 mission-first 结构被整体推翻后才能使用此卡片。

### 需求 8：系统应为后续驾驶舱与路线层提供统一输入

目的地卡片与目标摘要层应为以下后续能力提供统一输入：

- 路线推荐
- 接管点触发
- 驾驶状态解释
- 复盘与证据链说明
