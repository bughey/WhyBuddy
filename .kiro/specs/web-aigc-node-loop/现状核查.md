# web-aigc-node-loop 现状核查

## 结论

本轮已把 `web-aigc-node-loop` 收敛到“基于现有 `loop` edge 的最小可运行闭环”：

- `server/core/workflow-runtime-engine.ts` 在命中 `kind = "loop"` 的 edge 时，继续发射 `edge.loop_iterated`，并把迭代索引写入 `runtimeLoopIterations`；
- 同一路径新增了基于 edge `metadata` 的最小安全边界：
  - `maxIterations`
  - `maxDurationMs`
- 当 loop 超限时，runtime 会复用现有 `instance.terminated` 控制面进入 `FORCE_TERMINATED`，而不是引入第二套终止机制；
- 人工 `terminate(...)` 也会保留最近一次 loop 的上下文，使“loop 执行路径已接入 terminate 控制面”这件事有状态与事件证据；
- `server/tests/workflow-runtime-engine.test.ts` 与 `server/tests/web-aigc-runtime-observability.test.ts` 已补齐最小闭环测试证据。

但这仍然不是完整的 `loop` 节点产品化实现。当前完成的是“`loop` edge 最小闭环”，不是“正式 `loop` 节点 DSL / schema / 全量迭代模型”。

因此：

- `tasks.md` 中第 2、3、4 项现在可以按代码事实勾选；
- 第 1 项“定义循环类型与配置”仍不建议勾选。

## 分任务判断

### 1. 定义循环类型与配置

结论：暂不勾选。

已有证据：

- `shared/workflow-domain.ts`
  - `WEB_AIGC_EDGE_KINDS` 已包含 `"loop"`；
  - `WebAigcEdgeSchema` 已允许 `loop` 作为边种类。

仍未完成：

- `server/core/web-aigc-controlflow.ts` 里没有正式 `loop` 节点类型；
- shared contract 里没有 `loop` 节点专属 config schema；
- 还没有固定次数循环、条件循环、列表迭代等节点级配置模型；
- 本轮新增的 `maxIterations` / `maxDurationMs` 是读取 `loop` edge `metadata`，不是正式 schema。

判断依据：

- 当前实现属于“复用现有 `loop` edge 的最小治理”，不是“定义完成的 `loop` 节点能力模型”。

### 2. 增加次数与耗时保护

结论：可以勾选，但边界是“edge 级最小实现”。

已落地事实：

- `server/core/workflow-runtime-engine.ts`
  - 在 `transitionKind === "loop"` 路径读取 edge `metadata.maxIterations`；
  - 在同一路径读取 edge `metadata.maxDurationMs`；
  - 超限后会阻断该 edge，并复用现有 `instance.terminated` 事件与 `FORCE_TERMINATED` 状态；
  - loop 终止信息会写入：
    - `runtimeLoopTermination`
    - `runtimeTermination`
    - `runtimeLoopTracking`
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证超次数终止；
  - 已验证超耗时终止。

当前边界：

- 限制定义在 edge `metadata`，不是节点 schema；
- 没有独立的 loop 恢复策略、降级策略或更细的中断策略；
- 没有每轮完整输入/输出快照，仅保留最小迭代与中断摘要。

### 3. 写入迭代级回放事件

结论：可以勾选。

已落地事实：

- `server/core/workflow-runtime-engine.ts`
  - 实际发射 `edge.loop_iterated`；
  - metadata 至少包含：
    - `kind`
    - `loopKey`
    - `iterationIndex`
- `server/core/web-aigc-runtime-observability.ts`
  - 已把 `edge.loop_iterated` 映射成 replay `MILESTONE_REACHED`；
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证 `edge.loop_iterated` 事件与 `runtimeLoopIterations` 写入；
- `server/tests/web-aigc-runtime-observability.test.ts`
  - 已验证 `edge.loop_iterated -> replay milestone` 的桥接。

当前边界：

- 仍然没有完整的 graph projection / monitoring 侧 loop 轨迹可视化模型；
- 当前 replay 证据聚焦在 runtime 事件镜像，不等于完整前台回放产品闭环。

### 4. 接入人工终止与超时处理

结论：可以勾选，但边界是“复用现有实例级控制面”。

已落地事实：

- `server/core/workflow-runtime-engine.ts`
  - loop 超次数/超耗时时，不新建特殊控制面，而是复用现有 `terminateState(...) -> instance.terminated`；
  - `terminate(...)` 人工终止时，会把最近一次 loop 上下文写入 `runtimeTermination.loop` 和 `instance.terminated.metadata.loop`；
- `server/tests/workflow-runtime-engine.test.ts`
  - 已验证 loop guard 超限会终止；
  - 已验证人工 terminate 会保留最近 loop 上下文；
- `server/tests/web-aigc-runtime-observability.test.ts`
  - 已验证终止事件中的 loop metadata 能进入 replay 镜像数据。

当前边界：

- 还没有真正的“迭代执行中异步打断”机制；当前仍是单线程、步进式 runtime；
- `maxDurationMs` 是在每次即将再次进入 loop edge 时检查累计耗时，不是外部 watchdog 异步抢占；
- 没有 loop 专用 resume / continue / skip-iteration 控制面。
