# 需求文档：路线规划器与路线模型

## 目标

本 spec 定义“任务自动驾驶”中的 Route 对象与路线规划机制，使用户输入的 Destination 能被系统转换为可解释、可选择、可执行、可接管的任务路线。

Route 不是替代现有 `mission / workflow / runtime` 的底层实现，而是覆盖在其上的产品层与编排层抽象：

- 对用户来说，Route 是“系统准备如何把任务送达结果”的可视化路线。
- 对平台来说，Route 是把 Destination 映射到 Mission Runtime、Workflow 阶段、Agent 编队与治理策略的计划对象。
- 对运行时来说，Route 是可被执行、暂停、重试、重规划、回放与审计的路线快照。

## 背景

Cube Pets Office 当前已经具备以下基础：

- mission-first 的任务启动与执行语义；
- 十阶段工作流 pipeline；
- Web-AIGC 节点编排与运行时节点适配器；
- `WAITING_INPUT -> resume()` 的人工恢复链路；
- `terminate / retry / escalate` 的最小运行时控制面；
- replay / audit / telemetry / evidence 的可观察底座。

但在产品表达上，用户仍然更容易看到“任务、节点、执行状态”，而不是“目标、路线、风险、接管点”。本 spec 要补齐的就是 Route 这一层。

## 核心术语

### Destination

用户输入后被系统理解出的目标对象，包含目标、约束、成功标准、缺失信息与预期交付物。

### Route

系统为完成 Destination 生成的路线计划，包含阶段、步骤、并行/串行关系、推荐执行模式、风险点、接管点、预估成本与映射到 runtime 的执行依据。

### 主路线

系统默认推荐执行的路线。主路线应基于目标类型、上下文完整度、风险、成本、用户偏好与现有能力自动选择。

### 候选路线

系统给出的替代路线。候选路线用于支持“快速 / 标准 / 深度”等不同执行策略，而不是让用户直接编辑底层 DAG。

### 路线阶段

Route 内部用于表达任务推进的高层阶段，例如理解、澄清、规划、编队、执行、复核、交付。

### 路线步骤

Route 中可执行、可展示、可审计的最小产品层步骤。一个路线步骤可以映射到一个 workflow phase、一个 runtime node、一个 agent action，或一组并行子任务。

### 风险点

Route 中可能导致失败、偏航、成本超限、权限不足、结果不可信或需要人工判断的位置。

### 接管点

Route 中必须或建议用户介入的位置，包括澄清、审批、预算确认、权限确认、路线选择、结果验收与风险接受。

## 需求

### 需求 1：Route 对象必须成为一等计划对象

系统应定义独立的 Route 对象，用于描述从 Destination 到执行交付的路线计划。

验收标准：

- Route 必须包含唯一标识、关联 Destination、路线模式、路线阶段、路线步骤、风险点、接管点与运行时映射信息。
- Route 必须能表达“计划态”和“执行态”的差异，避免把用户可见路线直接等同于底层 workflow 图。
- Route 必须可序列化，便于持久化、回放、审计与前端展示。
- Route 必须保留生成原因，说明为什么推荐该路线。

### 需求 2：系统必须支持主路线与候选路线

系统应在规划阶段生成一条主路线，并在合适场景下生成多条候选路线。

验收标准：

- 主路线必须有明确推荐理由。
- 候选路线必须能解释与主路线的差异，包括速度、成本、质量、风险、自动化程度与接管次数。
- 用户可以在执行前选择候选路线，也可以在执行中通过重规划切换路线。
- 未被选择的候选路线应保留为规划证据，而不是直接丢弃。

### 需求 3：系统必须支持快速 / 标准 / 深度三类路线模式

Route Planner 应至少支持三种产品化路线模式。

验收标准：

- 快速路线：优先减少步骤与等待，适合快速原型、初稿、低风险任务。
- 标准路线：平衡质量、速度与可控性，作为默认路线模式。
- 深度路线：增加研究、复核、证据、审计与多轮修正，适合高风险、高价值或复杂任务。
- 三类路线必须能映射到不同的 workflow 阶段启用策略、Agent 编队规模、接管点密度与治理强度。

### 需求 4：Route 必须表达并行与串行安排

系统应在 Route 中显式表达哪些步骤必须串行执行，哪些步骤可以并行执行。

验收标准：

- 每个 Route Step 必须能声明依赖关系。
- Route 必须支持 parallel group，用于表达可并行执行的子任务集合。
- Route 必须支持 join / merge 语义，用于表达并行结果汇总点。
- 并行安排必须能映射到现有 Mission Runtime 或 Workflow Runtime 的执行约束。
- 当运行时暂不支持某些并行语义时，Route 必须能降级为串行执行并记录原因。

### 需求 5：Route 必须显式包含风险点

系统应在规划阶段识别并记录风险点。

验收标准：

- 风险点必须包含风险类型、影响范围、严重程度、触发条件、建议处理方式。
- 风险点类型至少包含上下文不足、权限不足、成本超限、结果质量不确定、外部工具失败、数据可信度不足、执行时间过长。
- 风险点应能影响路线推荐、接管点生成与 runtime 治理策略。
- 风险点应进入 replay / audit / telemetry 的可解释证据链。

### 需求 6：Route 必须显式包含接管点

系统应在路线中标出需要用户介入的位置。

验收标准：

- 接管点必须包含接管原因、触发条件、用户可选动作、默认动作、超时策略。
- 接管点类型至少包含澄清问题、路线选择、权限确认、预算确认、风险接受、结果验收。
- 接管点必须能映射到现有 HITL / decision / wait-resume 机制。
- 接管点应支持“必须接管”和“建议接管”两种强度。

### 需求 7：Route 必须能映射到现有 workflow 与 mission runtime

Route Planner 不应绕开现有执行系统，而应将 Route 映射到当前主仓已有能力。

验收标准：

- Route Stage 必须能映射到十阶段工作流 pipeline。
- Route Step 必须能映射到 workflow node、runtime adapter、agent action 或人工决策点。
- Route 的接管点必须能映射到 `WAITING_INPUT -> resume()`、`escalate()` 或现有人工审批链路。
- Route 的失败恢复必须能映射到 `retry / terminate / escalate` 控制面。
- Route 的执行过程必须能进入 replay / audit / telemetry。

### 需求 8：Route Planner 必须支持重规划

系统应在执行偏航、信息变化、工具失败或用户接管后支持重规划。

验收标准：

- 重规划必须保留原路线、变更原因、新路线与差异摘要。
- 重规划不得丢失已完成步骤的证据。
- 重规划可以选择继续当前路线、切换候选路线、生成新路线或请求用户接管。
- 重规划结果必须能被前端展示为“偏航与新路线”，而不是静默替换。

### 需求 9：Route 必须支持前端驾驶舱展示

Route 对象应服务于任务自动驾驶驾驶舱，而不仅是后端计划数据。

验收标准：

- 前端应能展示目的地、主路线、候选路线、当前阶段、当前步骤、风险点、接管点与剩余步骤。
- Route 应提供面向 UI 的摘要字段，避免前端直接解析底层 workflow 图。
- Route 应支持阶段进度、步骤状态、风险状态与接管状态的实时更新。
- Route 应支持用户理解“为什么走这条路线”。

### 需求 10：Route 生成必须可治理、可审计、可回放

Route Planner 的输出不能是一次性黑盒结果。

验收标准：

- Route 生成请求、输入上下文、推荐原因、候选路线与选择结果必须可追踪。
- Route 生成过程应记录关键决策原因。
- Route 的执行快照应能与 Mission Runtime 的事件流关联。
- 对高风险路线，系统必须保留风险接受与接管记录。
