# 需求文档：任务自动驾驶证据回放与信任链

## 目标

本 spec 定义任务自动驾驶中的证据链、回放链与信任链，使系统能够把一次自动驾驶任务从“理解目标、选择路线、执行步骤、请求接管、调用工具、复核结果、完成交付”的全过程，组织成一条可解释、可回放、可审计、可追溯的任务证据主链。

该主链需要同时回答以下问题：

- 任务是按什么驾驶时间线推进的。
- 哪些关键决策影响了路线和结果。
- 什么时候发生了路线变化、偏航修正或重规划。
- 哪些时刻交还给了用户或人工角色接管。
- 调用了哪些工具、访问了哪些资源、产生了哪些中间结果。
- 最终结果由哪些证据支持，是否经过 review / audit / verify / 验收。
- 如何把 audit、lineage、replay 串联起来形成可被信任的任务历史。

本 spec 的重点不是重新发明一套底层执行引擎，而是在现有 `mission / workflow / runtime / HITL / replay / audit / lineage` 能力之上，建立任务自动驾驶视角下的统一证据解释层。

## 背景

当前仓库已经逐步补齐了任务自动驾驶相关基础：

- `Destination / Route / Drive State / Fleet / Takeover Point / Replan` 的产品层抽象；
- Mission Runtime、workflow runtime 与节点执行事件；
- `waiting / decision / approval / resume / escalate` 等人工接管链路；
- replay、audit、lineage 等独立能力面；
- review / audit / verify / revise 等质量治理闭环。

但这些能力目前仍然偏分散：

- 回放可以讲“发生了什么”，但不一定能讲清“为什么这样发生”。
- 审计可以记录关键操作，但不一定能还原完整的任务驾驶上下文。
- 血缘可以追踪数据来源与流向，但不一定能直接解释路线与决策变化。
- 驾驶舱需要的是一条统一、连续、可解释的任务主链，而不是多个互相孤立的日志面板。

因此，需要新增“证据回放与信任链”这一层，把这些分散事实组织成同一条任务自动驾驶历史。

## 范围

本 spec 包含：

- 驾驶时间线的统一证据建模；
- 关键决策、路线变化、接管事件、工具调用、结果证据的统一建模；
- audit / lineage / replay 的串联关系；
- 面向信任链的完整性、来源、验证、解释要求；
- 与现有 Mission Runtime、Route、Drive State、Takeover 的兼容约束；
- 分阶段落地要求。

本 spec 不包含：

- 完整的回放 UI 布局与交互动效细节；
- 底层工作流引擎的整体重写；
- 所有工具调用协议的逐一改造；
- 不可篡改存储的完整密码学实现细节；
- 所有历史任务的一次性全量回填。

## 核心术语

### 驾驶时间线

面向任务自动驾驶主叙事的时间有序事件链，用于表达任务从开始到交付的高层推进过程。

### 证据项

对某个关键事实的结构化记录，既可服务回放，也可服务审计、血缘和结果解释。

### 证据链

由驾驶时间线作为主脊柱、各类证据项作为挂载节点形成的任务历史链路。

### 回放链

面向 replay 消费的有序事件视图，强调“何时发生、如何推进、用户看到什么”。

### 信任链

面向可信解释的验证链，强调“证据来自哪里、是否完整、是否可追溯、是否经过复核、是否足以支持结果”。

### 结果证据

用于支撑最终交付结果的摘要、工件、校验、复核、验收与来源引用集合。

## 需求

### 需求 1：系统必须定义统一的任务自动驾驶证据链模型

系统应把驾驶时间线、关键决策、路线变化、接管事件、工具调用、结果证据统一纳入同一组证据对象体系。

验收标准：

- 证据链必须至少覆盖 `drive_state_change`、`decision`、`route_change`、`takeover`、`tool_call`、`result` 六类关键证据。
- 每个证据项必须包含唯一标识、证据类型、发生时间、记录时间、摘要、来源角色、关联任务上下文。
- 每个证据项必须能关联 `mission / workflow / route / route step / runtime event` 中至少一层上下文。
- 证据链必须支持从单个证据项追溯其上游证据与下游结果。
- 证据链不得只由前端临时拼接，至少应可由服务端投影或事件层重建。

### 需求 2：系统必须定义统一的驾驶时间线

系统应把任务自动驾驶过程组织为一条面向用户可理解的驾驶时间线，作为所有证据串联的主脊柱。

验收标准：

- 驾驶时间线必须按时间顺序记录关键任务推进事件。
- 时间线至少应覆盖 `Drive State` 变化、路线启动、路线切换、接管触发、工具执行、结果提交和交付完成。
- 每条时间线事件必须说明前置状态、后置状态、触发原因、当前阶段和下一步预期动作。
- 时间线必须能表达中断、阻塞、重规划与恢复，不得只展示理想主链路。
- replay 至少应能基于该时间线重建关键阶段变化。

### 需求 3：系统必须记录关键决策证据

系统应把影响路线、执行和结果的关键决策记录为可解释证据，而不是只保留最终结论。

验收标准：

- 关键决策证据必须记录决策主题、决策输入、候选方案、最终选择、选择原因和置信信号。
- 决策证据必须能区分系统自动决策、人工决策和混合决策。
- 决策证据必须能关联触发该决策的时间线事件、路线步骤和相关风险。
- 若决策导致路线变化、接管或结果返工，系统必须能建立显式关联。
- replay 与 audit 必须都能消费关键决策证据，而不依赖各自维护单独副本。

### 需求 4：系统必须记录路线变化与重规划证据

系统应把路线切换、重规划、偏航修正视为一级证据对象，而不是普通日志噪音。

验收标准：

- 路线变化证据必须包含原路线、目标路线、触发原因、变化摘要和发起方。
- 路线变化证据必须能区分普通重试、步骤返工与真正的路线切换。
- 若已有步骤被保留、失效、跳过或新增，系统必须记录影响范围。
- 路线变化证据必须能关联对应的 `replanning` 或其他 Drive State 变化。
- 回放中必须能看到任务在哪个时刻从哪条路线切换到哪条路线。

### 需求 5：系统必须记录接管事件与人工恢复证据

系统应把所有接管请求、人工选择和恢复结果纳入统一证据链。

验收标准：

- 接管证据必须包含接管类型、触发原因、展示给用户的上下文、可选动作、默认动作和是否必需。
- 接管证据必须记录提交人、提交时间、所选动作、附加说明和超时处理结果。
- 接管证据必须能关联 `waiting / decision / approval / resume / escalate` 等现有链路。
- 若接管导致路线变化、继续执行、终止任务或降级执行，系统必须保留恢复结果。
- 审计与回放必须都能说明“为什么交还给用户”和“用户如何处理”。

### 需求 6：系统必须记录工具调用与执行证据

系统应把自动驾驶过程中的关键工具调用、资源访问与执行结果形成可解释证据。

验收标准：

- 工具调用证据必须包含工具标识、调用角色、调用目的、输入摘要、输出摘要、开始时间、结束时间和执行结果。
- 工具调用证据必须记录权限检查、成本信号、耗时、失败原因和副作用范围。
- 若调用涉及文件、外部 API、浏览器、沙箱、数据库或 MCP 工具，系统必须能标记资源类型。
- 工具调用证据必须支持敏感字段脱敏，不得为了审计而直接暴露全部原始数据。
- 若工具调用直接支撑某个决策或结果，系统必须能建立显式引用关系。

### 需求 7：系统必须定义结果证据与交付支撑链

系统应使最终结果能够被其上游证据解释和支撑，而不只是展示一个交付物链接。

验收标准：

- 结果证据必须包含结果摘要、工件引用、生成时间、生成路径和结果状态。
- 结果证据必须能关联支撑它的关键决策、工具调用、审计结论和血缘引用。
- 结果证据必须记录 review / audit / verify / revise / 验收 等质量信号。
- 若结果被驳回、返工或二次交付，系统必须能区分不同版本的结果证据。
- 驾驶舱、回放和审计都必须能从结果证据追溯到支撑它的证据链。

### 需求 8：系统必须串联 audit、lineage 与 replay

系统应把审计、血缘与回放纳入同一条任务证据主链，而不是并列的三个孤岛页面。

验收标准：

- 每个关键证据项必须支持关联 audit 记录、lineage 节点或 replay 事件引用中的至少一种。
- 同一条关键事件不得在 replay、audit、lineage 中出现彼此不一致的核心事实描述。
- 系统必须支持从 replay 事件跳转到相关 audit 记录和 lineage 视图。
- 系统必须支持从 audit 记录或 lineage 节点回溯到对应的驾驶时间线位置。
- 若某类证据暂时没有完整 audit 或 lineage 支持，系统必须显式标记缺口，而不是静默缺失。

### 需求 9：系统必须定义信任链与可信状态

系统应对证据来源、完整性、追溯性和验证状态进行统一表达，使结果“为什么可信”可以被解释。

验收标准：

- 每个关键证据项必须至少支持来源可信、链路完整、可回放、已复核中的一个或多个可信标记。
- 信任链必须能区分 `verified`、`partial`、`unverified`、`redacted` 等状态层级。
- 系统必须能说明某个结果证据的可信状态来自哪些上游证据与验证动作。
- 若证据存在缺失、脱敏、断链或未复核，信任链必须显式暴露风险，而不是仍显示为完全可信。
- 信任链不得只依赖 UI 文案推断，至少应有结构化状态可被服务端和回放消费。

### 需求 10：系统必须兼容现有自动驾驶与运行时模型

本 spec 应作为连接层，而不是要求现有自动驾驶和运行时能力整体重写。

验收标准：

- 不要求当前 Mission Runtime、workflow runtime 立即改名为证据链术语。
- 驾驶时间线必须能复用已有 `Drive State`、`Route`、`Takeover Point` 与 runtime event 作为输入事实。
- 接管证据必须兼容现有 `MissionDecision`、`submitMissionDecision()` 和 `resume()` / `escalate()` 链路。
- 路线变化证据必须兼容已有 Route Planner、Replan 和 Route Runtime Mapping。
- 结果证据必须兼容当前 review / audit / replay / lineage 能力的渐进接入。

### 需求 11：系统必须支持分阶段落地

证据链、回放链与信任链应允许逐步演进，而不是要求一次性改造全部存量能力。

验收标准：

- 第一阶段至少应完成术语统一、对象建模与文档约束。
- 第二阶段至少应完成驾驶时间线与关键证据的前端 view model 映射。
- 第三阶段至少应完成服务端事件投影、关联键与可重建的证据对象。
- 第四阶段至少应完成 replay / audit / lineage 的串联消费和信任状态展示。
- 文档不得把“一次性重构所有 runtime”作为前提条件。

## 验收标准

- 存在一份中文需求文档，明确说明任务自动驾驶证据链、回放链与信任链的目标、范围和边界。
- 文档中明确覆盖驾驶时间线、关键决策、路线变化、接管事件、工具调用、结果证据六类核心对象。
- 文档中明确要求 audit / lineage / replay 的串联关系，而不是分别描述三个孤立子系统。
- 文档中明确要求结果可信状态来自结构化信任链，而不是仅靠人工阅读日志。
- 文档中明确保持与现有 Route、Drive State、Takeover、Mission Runtime 的兼容关系。
- 后续实现任务可以基于本 spec 直接拆分为文档、投影层、服务端、回放面和审计/血缘接线工作项。
