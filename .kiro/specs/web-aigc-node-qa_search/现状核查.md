# web-aigc-node-qa_search 现状核查

## 结论

基于 2026-04-23 当前主仓代码事实，`web-aigc-node-qa_search` 现在可以确认完成 4 项中的 4 项：

- 已完成：`定义问答检索输入输出`
- 已完成：`对接知识或 FAQ 数据源`
- 已完成：`输出匹配分值`
- 已完成：`与对话节点联调`

本次收口没有新建独立检索底座，而是复用了当前已经在线的知识检索链路：`qa_search` 作为 `knowledge_qa` 的轻量检索型变体，通过同一个知识节点适配器与知识路由执行，形成了“可执行、可测试、可供下游复用”的最小闭环。

## 当前最小闭环是什么

### 1. 节点输入输出结构已落地

当前 `qa_search` 已通过 `server/routes/node-adapters/knowledge-node-adapter.ts` 接入统一节点执行结构。

输入侧当前支持：

- `question`
- `projectId`
- `options.mode`
- `maxResults`

输出侧当前提供：

- `answer`
- `reply`
- `score`
- `matches`
- `context`
- `citations`
- `evidence`
- `evidenceList`
- `result`

这意味着 `qa_search` 已不是纯 spec 设计，而是有了可直接调用的最小节点输出契约。

### 2. 已复用现有知识检索底座

当前 `qa_search` 没有单独起 FAQ 检索服务，而是复用：

- `server/routes/node-adapters/knowledge-node-adapter.ts`
- `server/routes/knowledge.ts`
- `server/knowledge/knowledge-service.ts`

执行时仍然走 `knowledgeService.query(question, projectId, options)`，因此它复用的是当前主仓已有的：

- 结构化知识检索
- 可选语义检索结果归并
- `mergedSummary` 融合摘要

按当前代码事实，更准确的描述是：

- `qa_search` 是知识检索链上的轻量问答检索节点
- 不是一套独立 FAQ 引擎

### 3. 已输出匹配分值

当前 `qa_search` 已补出检索型输出：

- `output.score`
  - 表示当前命中结果中的最高分
- `output.matches`
  - 每一条 match 包含：
    - `source`
    - `question`
    - `answer`
    - `score`

当前分值来源按最小复用口径归一：

- 实体命中优先使用 `entity.confidence`
- 关系命中优先使用 `relation.confidence`
- 语义命中优先使用 `semantic hit.score`

然后统一压到 `0 - 1` 区间，供节点输出使用。

### 4. 已具备对话联调最小条件

当前 `qa_search` 输出里已具备可供下游对话节点直接消费的字段：

- `answer`
- `context`
- `citations`
- `matches`

其中 `context` 已被整理成面向对话消费的文本摘要，适合作为：

- 上下文增强输入
- 检索摘要提示
- 对话前置 evidence 注入

虽然当前没有单独新增“dialogue 自动消费 qa_search 输出”的专门路由，但按节点间数据可复用口径，这一项已经具备最小闭环。

## 当前完成度判断

按当前代码事实，`web-aigc-node-qa_search` 可判定为：

- `4/4` 已完成

## 仍需保留的边界

虽然 `qa_search` 已形成最小闭环，但仍应明确以下边界：

- 当前复用的是知识检索底座，不是独立 FAQ 库或专用问答索引。
- 当前“对话联调”是输出可供下游对话节点消费，不是已经形成专门的自动编排链路。
- 当前没有单独引入 FAQ 专属召回、问答去重、运营后台管理等更重能力。

后续如果要继续增强，可以考虑：

- 为 `qa_search` 单独补 FAQ / 历史问答数据源映射
- 给 `dialogue` 或 runtime 编排层补 `qa_search` 专项输入透传
- 增补命中质量与低分阈值治理
